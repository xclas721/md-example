# 架構一致性檢查表（ACS vs 3DSS）

> 目的：確保 ACS 與 3DSS **架構一致**、但不強迫**功能一致**。  
> 使用時機：每次重構提案、每次 PR、每次跨系統調整前後。

## 使用說明

- 一次檢查一個能力域（例如：Overview alert、DIMP alert、iframe redirect）。
- 逐項標記 `PASS / FAIL / N/A`，`FAIL` 必須附上修正計畫或例外說明。
- 若為功能差異（業務正常差異），請標記 `N/A` 並附文件連結（建議指向 `06-decisions-and-open-questions.md`）。

---

## A. 分層與模組邊界

| 檢查項 | ACS | 3DSS | 狀態 | 備註 |
|--------|-----|------|------|------|
| Controller 僅負責輸入驗證/轉呼叫，不堆疊業務邏輯 |  |  |  |  |
| Service 負責業務邏輯，無 UI/路由耦合 |  |  |  |  |
| Utils 只放純工具邏輯（無隱性依賴） |  |  |  |  |
| Aspect 僅處理權限與上下文，無報表業務邏輯 |  |  |  |  |
| 常數與設定（ApiConst/Properties）集中管理 |  |  |  |  |

## B. API 契約與命名

| 檢查項 | ACS | 3DSS | 狀態 | 備註 |
|--------|-----|------|------|------|
| 路徑命名規則一致（系統前綴可不同） |  |  |  |  |
| Request DTO 命名與欄位語意一致（issuer/requestor 允許差異） |  |  |  |  |
| 回傳型別策略一致（單值/時序/列表） |  |  |  |  |
| 錯誤回應與例外處理模式一致 |  |  |  |  |
| 變數 API 與報表 API 的職責分界清楚 |  |  |  |  |

## C. 時間與查詢上限

| 檢查項 | ACS | 3DSS | 狀態 | 備註 |
|--------|-----|------|------|------|
| 時間格式皆為 epoch ms（UTC） |  |  |  |  |
| 區間語意一致（`[from_time, to_time]`） |  |  |  |  |
| 預設 interval 規則一致（未傳時） |  |  |  |  |
| 聚合 API 限制輸出點數，不截斷原始交易 |  |  |  |  |
| 明細 API 有 row cap 且有截斷標記 |  |  |  |  |
| 特例時間邏輯（如 DDoS 2m）有文件明示 |  |  |  |  |

## D. 權限與驗證流程

| 檢查項 | ACS | 3DSS | 狀態 | 備註 |
|--------|-----|------|------|------|
| Alert API 可由 `X-API-Key` 呼叫（無人登入） |  |  |  |  |
| Dashboard API 使用 `hitrustSsoToken` + RBAC |  |  |  |  |
| API Key 與 Token 流程不混用 |  |  |  |  |
| `X-BANK-ID` / `X-REQUESTOR-ID` 差異已文件化 |  |  |  |  |
| 權限失敗回應行為一致（訊息/碼） |  |  |  |  |

## E. 前端整合流程（Vue -> backend -> iframe -> Grafana）

| 檢查項 | ACS FE | 3DSS FE | 狀態 | 備註 |
|--------|--------|---------|------|------|
| 路由頁能正確取得 backend 動態 URL |  |  |  |  |
| backend 回傳 payload 欄位完整（token/org/time/vars） |  |  |  |  |
| iframe 載入後 Grafana query 參數與預期一致 |  |  |  |  |
| 時間參數在前後端與 Grafana 三端一致 |  |  |  |  |
| 失敗情境（token/API key/timeout）處理一致 |  |  |  |  |
